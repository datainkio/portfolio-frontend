## Architecture Overview (Ignore At Your Own Peril)

The Figma integration consists of orchestrated services that fetch design tokens and write CSS files:

- `services/PaletteService.js` - Fetches colors from Figma → writes `styles/colors.css`
- `services/TypographyService.js` - Fetches text styles → writes `styles/typography/fontFamilies.css`
- `../scripts/fetchFigma.js` - Orchestrates the entire sync process with error handling

**CRITICAL DEPENDENCY**: All services require a valid `FIGMA_TOKEN` environment variable with Files:read scope. Without this, services fail silently and leave you with stale design tokens.

### PaletteService.js - Color Token Management

**Purpose**: Fetches color styles from Figma and generates CSS custom properties
**Input**: Figma file ID and color style hierarchy
**Output**: `styles/colors.css` with CSS custom properties
**Dependencies**: Figma API token, specific color naming structure

**CRITICAL FIGMA STRUCTURE REQUIREMENTS**:
```

Colors Frame
├── Primary/Light (Color style)
├── Primary/Dark (Color style)
├── Primary (Color style)
├── Secondary/Light (Color style)
└── Secondary/Dark (Color style)

````

**The service expects EXACTLY this naming convention**. Change "Primary/Light" to "Primary Light" and watch the system implode. The forward slash is not optional decoration - it's a parsing requirement.

**Generated CSS Structure**:
```css
/* Auto-generated by PaletteService.js - DO NOT EDIT MANUALLY */
:root {
  --color-primary-light: #hexvalue;
  --color-primary-dark: #hexvalue;
  --color-primary: #hexvalue;
}
````

**LETHAL WARNING**: This file is completely overwritten on every sync. Manual edits will vanish faster than your patience when debugging CSS cascade issues.

### TypographyService.js - Font Family Generation

**Purpose**: Fetches text styles from Figma and generates font-family utilities
**Input**: Figma text styles with font family information
**Output**: `styles/typography/fontFamilies.css` with Tailwind-compatible utilities
**Dependencies**: Figma API token, text style hierarchy

**REQUIRED FIGMA TEXT STYLE STRUCTURE**:

```
Typography Frame
├── Heading/Large (Text style with font-family)
├── Heading/Medium (Text style with font-family)
├── Body/Regular (Text style with font-family)
└── Caption/Small (Text style with font-family)
```

**Generated Font Family Utilities**:

```css
/* Auto-generated by TypographyService.js - DO NOT EDIT MANUALLY */
.font-heading {
  font-family: "Your Heading Font", system-ui, sans-serif;
}
.font-body {
  font-family: "Your Body Font", system-ui, sans-serif;
}
.font-caption {
  font-family: "Your Caption Font", system-ui, sans-serif;
}
```

**DEPENDENCY WARNING**: The service extracts font-family values from Figma text styles. If you change fonts in Figma without running the sync, your CSS will reference fonts that don't exist, causing fallback font chaos.

## API Integration Patterns (Follow These Or Face API Hell)

### Authentication & Headers

All services use consistent authentication patterns:

```javascript
const headers = {
  "X-Figma-Token": process.env.FIGMA_TOKEN,
  "Content-Type": "application/json",
};
```

**SECURITY CRITICAL**: The Figma token provides full read access to your design files. Guard it like nuclear launch codes. Never commit it to version control.

### Error Handling Patterns

Services implement defensive error handling for API failures:

```javascript
try {
  const response = await fetch(figmaUrl, { headers });
  if (!response.ok) {
    throw new Error(
      `Figma API failed: ${response.status} ${response.statusText}`,
    );
  }
  // Process successful response
} catch (error) {
  console.error("Service failed:", error.message);
  // Continue with existing files rather than breaking build
}
```

**FAIL-SAFE BEHAVIOR**: Services are designed to fail gracefully. If Figma API is down, services log errors but don't break the build. This means you might be working with stale design tokens without realizing it.
